Chat Card — Complete Lifecycle

The chat card is the interactive resolution workspace. One card handles the entire flow: roll result, defender reaction, damage application, and feedback. These mockups show the full picture for both player and GM perspectives.

These wireframes show information architecture and interaction flow, not visual design. Final colors, typography, spacing, and component styling will differ.
Key Nimble mechanic: There is no separate attack roll. The damage dice ARE the attack — the Primary Die (leftmost) determines hit (not 1), miss (1), or crit (max). The total of all dice + modifier = damage on a hit.

Chat Card Design Principles
One card, full resolution: No secondary cards generated. The entire attack → reaction → damage chain lives on one card.
Progressive state: Card evolves — roll result → Defend decision → Interpose → damage feedback. Each layer adds info without removing previous context.
Role-gated: Players see Defend/Take It buttons. GMs see Apply Damage + target selector. Neither sees the other's actions.
Informed decisions: Defend prompt shows exact math. Action cost stated. Players make tactical trade-offs, not blind choices.
Trust feedback: HP before → after shown on every application. Players never wonder "did the system do that correctly?"
Reversible: Every Apply action (damage and healing) has an Undo button that restores HP to snapshot values. Mistakes cost seconds, not minutes.
Effect node tree: Cards use a composable effect tree (damage, healing, savingThrow, condition, note). Saving throws branch into pass/fail with different child effects. Conditions render as clickable pills. This tree structure handles complex spells with multiple conditional outcomes.

Layout Rationale
Stats bar: Always visible, never scrolls. HP, mana, armor (damage reduction via Defend reaction), speed — the key values a player checks most.
Resource bars: Visual HP/mana at a glance. Reinforces the numeric values above with a spatial representation.
Action economy: Nimble heroes have a base of 3 actions per turn, but this can vary — initiative rolls, class features, and effects can grant more or fewer. Reactions (Defend, Opportunity Attack, Interpose, Help) cost 1 action each and pre-spend from your next turn. Pips are dynamic: filled = available, faded = used. Count adjusts to match actual total.
Conditions: Active effects visible without opening a sub-menu. Color-coded (debuff/buff) with duration.
Ability list: Each row is a single click to activate. Name, cost, range, type — enough to decide, not enough to overwhelm. "Cast" / "Attack" button is the activation trigger.
Lock button: Prevents accidental edits during play. Unlocked mode allows manual stat adjustment.

Layout Rationale
Active turn highlight: Left border accent + background tint. GM always knows whose turn it is without checking the carousel.
Inline actions: Attack buttons right on the row. One click to initiate — no opening sheets. This is the "glance, decide, act" principle in action.
HP visualization: Numeric + tiny bar. Color shifts from green → yellow → red as HP decreases. GM sees health state instantly.
Synergy reminders: Yellow inline notes surface relevant buffs/abilities from other creatures. GM doesn't need to remember — the tracker reminds.
Dead creatures: Faded but not hidden. GM can still see them for reference.
Minion groups: Collapsed into a single row with group count. "Group Attack" resolves all at once. "Split" divides into sub-groups for targeting different players.
Expanded stats: Only the active creature auto-expands. Click any row to toggle stats. Keeps the default view ultra-compact.

Existing Implementation Notes
Roll Mode Slider: Uses svelte-range-slider-pips library. Range -6 to +6 with labels: "Disadvantage ×N", "Straight Roll", "Advantage ×N". Shared RollModeConfig component used by both dialogs.
Situational Modifiers: Accepts formula strings (not just numbers) — validated with Roll.validate(). This supports expressions like "1d4" or "+2".
Primary Die: Two controls — "Set Primary Die" overrides the value entirely, "Primary Die Modifier" adjusts it. Both are number inputs.
Damage Formulas: Displayed with damage type labels (e.g., "Fire: 2d6+3"). Formulas are computed from actor roll data and show the actual values.
Sensible defaults: Straight roll, no modifiers, no primary die override. Common case = just click Roll.
Layout pattern: nimble-sheet__body + nimble-sheet__footer. Full-width Roll button in footer. All labels localized via CONFIG.NIMBLE.
