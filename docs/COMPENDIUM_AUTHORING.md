# Compendium Authoring

Reference for writing pack JSON files, understanding the folder layout, and using build commands.

---

## Pack Source Folder Layout

```
packs/
  ids.json                          # Stable ID registry (auto-managed, do not edit)
  <packName>/
    core/
      <subgroup>/
        <entry-name>.json
```

Examples:
- `packs/monsters/core/bandits/bandit-assassin.json`
- `packs/spells/core/fire/flame-dart.json`
- `packs/boons/core/major/tough.json`

The `subgroup` subdirectory is a freeform organizational folder — it does not affect the compiled pack.

---

## Pack Types

All packs are declared in `dist/system.json` under the `"packs"` array.

| Pack name (source dir) | `type` in system.json | Document types stored |
|---|---|---|
| `monsters` | `Actor` | `npc` actors |
| `legendaryMonsters` | `Actor` | `soloMonster` actors |
| `spells` | `Item` | `spell` items |
| `boons` | `Item` | `boon` items |
| `ancestries` | `Item` | `ancestry` items |
| `backgrounds` | `Item` | `background` items |
| `classes` | `Item` | `class` items |
| `classFeatures` | `Item` | `feature`, `subclass` items |
| `subclasses` | `Item` | `subclass` items |
| `items` | `Item` | `object` items |
| `magicItems` | `Item` | `object` items |
| `tables` | `RollTable` | Roll tables |

---

## Build Commands

```bash
pnpm build:compendia   # Compile pack JSON -> LevelDB .db files in dist/packs/
pnpm build:system      # Build TypeScript/Svelte only
pnpm build             # Full build: compendia + system
```

The compiled `.db` files land at `dist/packs/<packName>.db` and are served by FoundryVTT.

---

## `_id` Field and `ids.json`

- **Omit `_id`** when creating a new entry. The build pipeline assigns a stable ID on first compile and writes it to `packs/ids.json`.
- Once an entry has an ID in `ids.json`, subsequent builds reuse the same ID, keeping compendium links stable.
- Embedded items inside an actor (e.g., monster features inside an NPC) **do** carry their own `_id` directly in the JSON, because they are not independently tracked in `ids.json`.

## NPC / Monster Actor JSON Schema

Minimal working example (`type: "npc"` or `type: "soloMonster"`):

```json
{
  "name": "Bandit Assassin",
  "type": "npc",
  "folder": "718123f1f5a7d7b3",
  "img": "icons/svg/mystery-man.svg",
  "system": {
    "attributes": {
      "armor": "none",
      "damageResistances": [],
      "damageVulnerabilities": [],
      "damageImmunities": [],
      "hp": {
        "max": 24,
        "temp": 0,
        "value": 24
      },
      "sizeCategory": "medium"
    },
    "description": "",
    "details": {
      "creatureType": "Bandits",
      "isFlunky": false,
      "level": "2"
    },
    "savingThrows": {
      "strength":     { "defaultRollMode": 0, "mod": 0 },
      "dexterity":    { "defaultRollMode": 0, "mod": 0 },
      "intelligence": { "defaultRollMode": 0, "mod": 0 },
      "will":         { "defaultRollMode": 0, "mod": 0 }
    }
  },
  "prototypeToken": {
    "name": "Bandit Assassin",
    "displayName": 50,
    "actorLink": false,
    "width": 1,
    "height": 1,
    "disposition": -1,
    "displayBars": 40,
    "bar1": { "attribute": "attributes.hp" },
    "bar2": { "attribute": null }
  },
  "items": [],
  "effects": [],
  "flags": {}
}
```

Key fields:
- `folder` — references a Folder `_id`; use `null` if not organizing into folders
- `system.details.level` — stored as a string
- `system.attributes.armor` — `"none"`, `"light"`, `"medium"`, `"heavy"`, or a formula string
- `prototypeToken.displayName: 50` — hover to show name; `30` = always show
- `prototypeToken.disposition: -1` — hostile; `0` = neutral; `1` = friendly
- `_stats` — auto-generated by FoundryVTT; safe to omit in source files

## Monster Feature (Embedded Item) Schema

Monster features live inside the actor's `"items"` array. They have type `"monsterFeature"`.

```json
{
  "name": "Poison Blade (2x).",
  "type": "monsterFeature",
  "_id": "E60UdXxj5NhOhoKP",
  "img": "icons/weapons/daggers/dagger-poisoned.webp",
  "system": {
    "macro": "",
    "identifier": "",
    "rules": [],
    "activation": {
      "acquireTargetsFromTemplate": false,
      "cost": {
        "details": "",
        "quantity": 1,
        "type": "action",
        "isReaction": false
      },
      "duration": { "details": "", "quantity": 1, "type": "none" },
      "effects": [
        {
          "id": "vgJWrDKCn1ZhdF6j",
          "type": "damage",
          "damageType": "piercing",
          "formula": "1d8+2",
          "parentContext": null,
          "parentNode": null,
          "canCrit": true,
          "canMiss": true,
          "on": {
            "hit": [
              {
                "id": "x1NTHvj33BiSHrLS",
                "type": "condition",
                "condition": "dazed",
                "parentContext": "hit",
                "parentNode": "vgJWrDKCn1ZhdF6j"
              },
              {
                "id": "qaPQfrh0bJHyYRYP",
                "type": "damageOutcome",
                "outcome": "fullDamage",
                "parentContext": "hit",
                "parentNode": "vgJWrDKCn1ZhdF6j"
              }
            ]
          }
        }
      ],
      "showDescription": true,
      "targets": { "count": 1, "restrictions": "", "attackType": "", "distance": 1 },
      "template": { "length": 1, "radius": 1, "shape": "", "width": 1 }
    },
    "description": "<p>1d8+2. On damage: Dazed.</p>",
    "subtype": "action"
  },
  "effects": [],
  "folder": null,
  "sort": 0,
  "flags": {}
}
```

### Effect `on` contexts

Each damage effect node can have child nodes under `on.hit`, `on.miss`, `on.criticalHit`. Common child types:

| `type` | Purpose | Key fields |
|---|---|---|
| `"damage"` | Deal damage | `damageType`, `formula`, `canCrit`, `canMiss` |
| `"condition"` | Apply a condition | `condition` (condition ID string) |
| `"damageOutcome"` | Specify damage resolution | `outcome: "fullDamage"` \| `"halfDamage"` |
| `"note"` | Display a note | `noteType: "warning"`, `text` |

## Spell Item JSON Schema

```json
{
  "name": "Flame Dart",
  "type": "spell",
  "img": "icons/magic/fire/projectile-bolt-zigzag-orange.webp",
  "system": {
    "macro": "",
    "identifier": "",
    "rules": [],
    "activation": {
      "cost": { "quantity": 1, "type": "action", "isReaction": false },
      "effects": [ /* same effect nodes as monster features */ ],
      "targets": { "count": 1, "restrictions": "" },
      "template": { "shape": "", "length": 1, "radius": 1, "width": 1 }
    },
    "description": {
      "baseEffect": "<p><strong>1 Action, Single Target</strong></p><p>...</p>",
      "higherLevelEffect": "<p>+5 damage every 5 levels.</p>",
      "upcastEffect": ""
    },
    "properties": {
      "range": { "min": 2, "max": 8 },
      "reach": { "min": 1, "max": null },
      "selected": ["range"]
    },
    "school": "fire",
    "tier": 0
  },
  "effects": [],
  "folder": null,
  "flags": {}
}
```

Spell-specific fields:
- `system.school` — spell school string (e.g., `"fire"`, `"wind"`, `"radiant"`, `"necrotic"`, `"ice"`, `"lightning"`, `"utility"`, `"secret"`)
- `system.tier` — spell tier (0 = novice, higher = more powerful)
- `system.description.baseEffect` — main HTML description shown in sheet/chat
- `system.description.higherLevelEffect` — HTML shown for scaling info
- `system.description.upcastEffect` — HTML for upcast rules (if any)

## Adding a New Pack Type

1. Create the source folder: `packs/<newPackName>/core/`
2. Add an entry to the `"packs"` array in `dist/system.json`:

```json
{
  "name": "nimble-<new-pack-name>",
  "label": "Nimble New Pack Label",
  "path": "packs/<newPackName>.db",
  "type": "Item",
  "system": "nimble"
}
```

3. Run `pnpm build:compendia` — the build script picks up any folder matching an entry in `system.json`.

## Tips

- The `_stats` block (`coreVersion`, `systemId`, `systemVersion`, `createdTime`, `modifiedTime`, `lastModifiedBy`) is auto-generated when documents are saved in Foundry. It's safe to include or omit in source files; the build process does not rely on it.
- The `folder` field on a top-level actor/item references a Folder document `_id`. Use `null` to leave unorganized.
- Embedded item `_id` values must be unique within their parent actor's `items` array. Use a random 16-character alphanumeric string.
- Roll formulas support `@level`, `@strength`, `@dexterity`, etc. as data references (resolved at activation time against the caster's roll data).
